# `.reinguard` (Semantics layer)

Repository-local **semantic control plane** for `rgd`: substrate config,
knowledge atoms, policy, and control (state / route / guard) match rules.
See **ADR-0001** (Adapter / Semantics / Substrate), **ADR-0011** (this
layout), and **ADR-0010** (knowledge).

Normative architecture lives in `docs/adr/`. This directory describes how
*this* repository configures `rgd`.

## Layout

| Path | Role |
|------|------|
| `reinguard.yaml` | `schema_version`, `default_branch`, `providers` (ADR-0008, ADR-0009) |
| `knowledge/` | Knowledge Markdown + YAML front matter; `manifest.json` generated by `rgd knowledge index` |
| `policy/` | Normative documents (not indexed by `knowledge index`) |
| `control/states/*.yaml` | `type: state` rules only |
| `control/routes/*.yaml` | `type: route` rules only |
| `control/guards/*.yaml` | `type: guard` rules only |

Unified **priority** across all control YAML files: ADR-0004.

## Where to put new content

- **Improves judgment when read** → `knowledge/` (front matter required; run `rgd knowledge index`).
- **Must be followed** (review/merge invariants, etc.) → `policy/`.
- **Evaluable match rules** → `control/` in the subtree matching `type`.
- **Client-specific procedures** → `.cursor/` (Adapter layer); do not duplicate policy body text there.

## Authoring knowledge

Each file under `knowledge/` is Markdown with a **YAML front matter** block
(delimited by `---`).

### Required front matter

| Field | Rule |
|-------|------|
| `id` | Unique across entries |
| `description` | One-line summary |
| `triggers` | Non-empty keyword list for `rgd knowledge pack --query` |

### After editing knowledge metadata

1. `rgd knowledge index`
2. `rgd config validate`
3. Commit the `.md` and `manifest.json`

See [`knowledge--ops.md`](knowledge/knowledge--ops.md) for retrieval flow.

## File naming (knowledge)

Pattern: **`<prefix>--<topic>.md`** (`--` separates prefix from topic).

Prefixes align with `rgd` areas (see the **Layout** table above): `observation--`,
`resolve--`, `guard--`, `knowledge--`, `config--`, `schema--`, `cli--`,
`match--`, plus cross-cutting `testing--`, `ci--`, `review--` for operational
references that remain knowledge (not policy).
